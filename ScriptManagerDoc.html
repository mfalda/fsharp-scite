      If you haven't noticed already, I try to keep everything structured and organized.  In addition,
  the documentation of some files is longer than the code itself (see UpdateStatusBar.lua). However, I
  feel this enables you to fully understand what I am trying to accomplish, and how I am doing it.
 
      Before you read the documentation, I would like to point out that I am not trying to eliminate

  Extman, in fact, if you've never used Extman, I really suggest you check it out.  My reason for

  creating Script Manager was to improve Extman by adding different elements and ideas, as you will see

  in the documentation.  Also, I wanted to make a very user—friendly script manager.  Finally, without

  Extman, I would not have been able to create Script Manager.

  | 1) Introduction |
  \—————————————————————————————————————————————————————————————————————————————————————————————————————
      Script Manager is a lua script manager for SciTE (just like Extman).  It enables multiple scripts
  to capture standard (and custom) events without interfering with other scripts.  Like Extman, it can
  also define commands without changing the property files.
 
      Script Manager was created in SciTE v1.75 on Windows ME (yes, I have a Windows ME and I hate it).
  Because I have Windows ME I could not update to SciTE v1.76.  Despite this, I believe that it will
  work on previous and later versions of SciTE, except for one element (see GetLexerName).  Extman is
  available on Windows and GTK, however, because I don't have access to it, Script Manager does not
  support GTK, however, I would be happy to make it compatible if I had suggestions from some GTK users.
  In addition, Script Manager does not support SciteOther, as I don't feel any need for it, just have
  the DOS window start minimized.  Finally, Script Manager is compatible is Extman scripts.

  | 2) Installation & Uninstallation |
  \—————————————————————————————————————————————————————————————————————————————————————————————————————
      Installing and uninstalling Script Manager is just as easy as it is with Extman.  Just place the
  file 'ScriptManager.lua' wherever you wish.  Then set the property 'ext.lua.startup.script' to
  that file.  All the Script Manager / Extman scripts within that directory and subdirectories will be
  loaded.  If you wish to change that directory, then set the custom property 'ext.lua.script.directory'
  (as compared to Extman's 'ext.lua.directory') to that directory.  To make your life easier, Script
  Manager will never load ScriptManager.lua or extman.lua.  In addition, because scripts, which are not
  compatible with Script Manager or Extman, cause problems (they override the event handlers) they will
  not be loaded into Script Manager.  Just so you know, Extman does not have either of these functions.
  If Script Manager doesn't load at first, just restart SciTE.
 
      To uninstall Script Manager (why would you do that?) just get rid of the 'ext.lua.startup.script'
  property.  You could move the location of Script Manager, but that will cause an error.  If you leave
  the 'ext.lua.script.directory' property, it will have no effect in SciTE.

  | 3) Starting Script Manager |
  \—————————————————————————————————————————————————————————————————————————————————————————————————————
      There are a few things that you need to know about Script Manager's startup.  Like Extman, on
  startup, it obtains all the files within the specified directory; this is where you see the
  'flashing black box' called DOS (they should make os.execute less system dependent).  After this is
  complete, Script Manager does some things different than Extman.  First, it sets the 'ext.lua.reset'
  property to 0 (this means that SciTE will only check Script Manager once — on startup), this helps
  if you have the property set to 1, because the DOS window will keep showing up.  However, alone, this
  can be a pain, like if you were put a new script into the Script Manager directory when SciTE is open,
  you would have to close and reopen SciTE.  So, to help you out, I added a function, "Refresh Manager"
  that's like Extman's "Reload Script" but global.  If has the same effect of 'ext.lua.reset' being 1
  and a buffer switch, (except, it's at the click of a button).  The final thing to know about the
  startup is that if you haven't defined the 'dwell.period' property (if you don't know what it is don't
  bother looking in SciTEDoc.html, it's undocumented) Script Manager sets it to 400.  Without setting
  this property, SciTE's OnDwellStart event handler won't work.

  | 4) Glitches Removed from Extman |
  \—————————————————————————————————————————————————————————————————————————————————————————————————————
      Extman is a very successful script manager, but it still has some glitches (at least, I've
  experienced them — remember Windows ME).  One of the most prominent glitch I've seen is in the Tools
  menu (in fact, there are a few).  When the buffer is switched, sometimes functions are removed from
  the menu for no reason (the file pattern is the same of course), or functions are repeated, I've seen
  "Reload Script" duplicated three times!  Also, Extman doesn't remove functions that don't exist
  anymore, say you deleted a file while SciTE was open.  Script Manager fixes these problems and adds
  new functions.  On "Refresh Manager", Script Manager removes any non—existent commands.  Also, if a
  function selected doesn't exist, it is removed as well.  In Extman, if a shortcut already exists, the
  new command doesn't receive a shortcut.  This still holds true in Script Manager, however, if the
  command with the shortcut is removed (doesn't exist) the other command gets it shortcut back.  In fact
  an entire list is created, so if the new function doesn't exist, the next function gets the shortcut.
 
      When a script is required in Extman it is run right then an there, and supposedly doesn't run
  again.  However, I found that it did run again.  Try it for yourself (using print). This is fixed in
  Script Manager.  It is well known that Extman doesn't support multiple list types for user lists, but
  Script Manager does, in fact, I didn't find a limit to the number of list types (I went close to 100).
  Extman's scite_OnWord event handler returns the word that was inputted into SciTE, however, if there
  was a backspace, the deleted character or characters are not removed, in Script Manager, however, they
  are deleted.  Finally, when a script returns other than false in Extman (such as prompt.lua) or if
  there is an error, the rest of the events don't run, of course, this is not true for Script Manager.
  Another bonus you get with Script Manager is when a command is run, it is run in protected mode,
  which means that any error isn't sent to SciTE, halting the execution.  Instead, Script Manager
  captures the error, displays it like normal, but continues the rest of the functions.
 
      In addition to Extman's extended handlers (scite_OnEditorLine, scite_OnOutputLine, scite_OnWord),
  Script Manager includes new handlers.  These include OnDwellEnd, OnMarginDoubleClick, and OnStartUp.
  More on these later, this is just a heads up.  (Personally, I like OnMarginDoubleClick the most.)

  | 5) Adding Commands with Script Manager |
  \—————————————————————————————————————————————————————————————————————————————————————————————————————
      Adding commands to the Tools menu in Script Manager is just as easy as in Extman.  However, the
  function that creates the command is in more traditional form than Extman's.  Although scite_Command
  is still supported, I feel it is better to use SetCommand.  SetCommand is very flexible with its
  parameters, if you leave one out, then it is just set to a default value.  Remember, if a shortcut
  issued is already being used, the new command won't get a shortcut, but it will be backed up.

  | 6) Registering Events in Script Manager |
  \—————————————————————————————————————————————————————————————————————————————————————————————————————
      Similar to commands, registering events is very easy in Script Manager.  The same process applies
  as in Extman, however, new, optional, features have been added.  The first feature is an extension;
  like Extman, you can optionally have the event removed after use, however, now you can select how many
  times the event is used (really, I find no use for the remove parameter, but I decided to keep and
  upgrade it anyway).  The second feature is brand new, and I find it very useful.  Some functions
  should run before other functions, for example, a function that affects the SciTE window should run
  before a function that doesn't.  Therefore, I added the 'priority' parameter.  The priority can be set
  from 1 to 10 (1 being the highest priority).  Although I see no need for a priority level below 5, I
  decided to put it there.  In Extman, the event that was registered first runs first, the priority
  setting sorts it a little better (as long as you are reasonable about it).  But, events with the same
  priority will follow the first come first serve rule.  Finally, in order for the registration to work
  the event should be registered at the end of your script (it must be registered after the function).
